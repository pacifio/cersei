# LongMemEval — Cersei vs Mastra / Zep / Supermemory

[LongMemEval](https://arxiv.org/abs/2410.10813) (ICLR 2025) is the 500-question long-term-memory benchmark Mastra used for its [*Observational Memory*](https://mastra.ai/research/observational-memory) research, Zep reports on in the [Graphiti paper](https://arxiv.org/abs/2501.13956), and Supermemory cites in its comparisons. We run the same 500 questions through four Cersei memory configurations so the numbers line up one-for-one with what those frameworks publish.

## TL;DR

Ran `longmemeval_s` — 500 questions — on **2026-04-24** against four Cersei configurations. Answerer, judge and observer are all `gemini-2.5-flash`; `EmbeddingMemory` uses `gemini-embedding-001` (3072-d, Matryoshka). The Observer prompt is a [verbatim port](https://github.com/pacifio/cersei/blob/main/bench/long-mem/src/mastra_prompts.rs) of Mastra's `OBSERVER_EXTRACTION_INSTRUCTIONS`, and the answerer's system message wraps retrieved context with Mastra's `OBSERVATION_CONTEXT_PROMPT` + `OBSERVATION_CONTEXT_INSTRUCTIONS` (KNOWLEDGE UPDATES / PLANNED ACTIONS / MOST RECENT USER INPUT).

| Config | Overall (macro, excl abstention) | Abstention | Correct / total | Input tokens (sum) | Avg wall (ms) |
|---|---|---|---|---|---|
| **A. Full-context baseline** (`JsonlMemory`) | 84.6 % | 86.7 % (26/30) | 422 / 500 | **53.16 M** | 11 164 |
| **B. Semantic recall** (`EmbeddingMemory`, usearch HNSW + gemini-embedding-001) | 84.2 % | 86.7 % (26/30) | 429 / 500 | **2.68 M (20× fewer)** | 45 470 |
| **C. Graph substring** (`GraphMemory`, grafeo) | 6.6 % | **100 % (30/30)** | 46 / 500 | 0.24 M | 4 300 |
| **D. Hybrid** (Observer + embed + graph + RRF) | **85.7 %** | **93.3 % (28/30)** | **432 / 500** | **1.58 M (34× fewer)** | 167 000 |

**Headlines**

- **Cersei Hybrid @ 85.7 %** beats Supermemory on both `gemini-3-pro-preview` (85.2 %) and `gpt-5` (84.6 %), Mastra OM on `gpt-4o` (84.23 %), Mastra RAG on `gpt-4o` (80.05 %), Zep on `gpt-4o` (71.2 %), and lands within **0.3 pp** of EmergenceMem Internal on `gpt-4o` (86.0 %).
- The remaining gap to Mastra OM / `gemini-3-flash-preview` (89.2 %) and `gpt-5-mini` (94.87 %) is concentrated on answerer-model tier, not algorithm — same observational-memory pipeline, newer model. Porting to `gemini-3-flash-preview` once the Cersei Gemini provider supports it should close most of that.
- **Pure semantic retrieval (`EmbeddingMemory`, 84.2 %) already beats Mastra OM / `gpt-4o` (84.23 %) and Supermemory / `gpt-4o` (81.6 %) with 20× fewer input tokens than our own full-context baseline.** On this dataset, a well-tuned HNSW RAG is enough to clear most of the field.
- **Graph-substring (6.6 %)** is the honest floor — substring match can't paraphrase-match. It still hits 100 % abstention because it returns junk → the answerer correctly refuses.

### Cost comparison

| Config | Input tokens | Est. Gemini cost (`gemini-2.5-flash` @ $0.30/1M in, $2.50/1M out) | Avg wall / Q |
|---|---:|---:|---:|
| Baseline | 53.16 M | **$15.95** | 11.2 s |
| Embed | 2.68 M | **$0.80** + ~$0.05 embed = **$0.85** (~19× cheaper than baseline) | 45.5 s |
| Graph | 0.24 M | **$0.07** (228× cheaper than baseline) | 4.3 s |
| Hybrid | 1.58 M | **$0.47** + ~$5–7 observer = **$5–7** (~2–3× cheaper than baseline) | 167 s |

The per-question wall time difference between embed (45.5 s) and hybrid (167 s) is the cost of the Observer pass: hybrid runs `gemini-2.5-flash` once per haystack session (30–40 sessions per question on `longmemeval_s`) to extract structured observations before retrieval. The accuracy gain buys +1.5 pp over embed-only.

## The four configurations

| Config | Cersei backend | What it tests |
|---|---|---|
| **A. `baseline`** | `JsonlMemory` — full haystack in prompt | Control lower bound. The answerer sees every turn, capped only by the LLM context window. |
| **B. `embed`** | `EmbeddingMemory` (`usearch` HNSW + `gemini-embedding-001` at 3072-d, cosine) | Pure semantic retrieval. Directly comparable to Mastra RAG. |
| **C. `graph`** | `GraphMemory` (grafeo substring + query-word-overlap rerank) | Honest floor for the graph layer — no LLM extraction, no semantic matching. |
| **D. `hybrid`** | **LLM fact extractor → EmbeddingMemory + GraphMemory → RRF fusion** | The configuration that actually competes with Mastra's observational memory. |

## Methodology (non-negotiable)

- **Dataset:** [`xiaowu0162/longmemeval`](https://huggingface.co/datasets/xiaowu0162/longmemeval) — 500 questions, 6 question types (`single-session-user`, `single-session-assistant`, `single-session-preference`, `multi-session`, `temporal-reasoning`, `knowledge-update`) plus abstention cases (detected via `question_id.ends_with("_abs")`). Variants: `longmemeval_s` (~115k tokens/Q, headline), `longmemeval_m` (~1.5M tokens/Q), `longmemeval_oracle` (evidence-only control).
- **Judge rubric:** **verbatim port** of Mastra's six question-type prompts (`_inspirations/mastra/explorations/longmemeval/src/evaluation/longmemeval-metric.ts`), which in turn copies from the official Python evaluator.
- **Observer rubric:** **verbatim port** of [`OBSERVER_EXTRACTION_INSTRUCTIONS`](https://github.com/pacifio/cersei/blob/main/bench/long-mem/src/mastra_prompts.rs) from Mastra's `@mastra/memory` — temporal anchoring, user-assertions-beat-questions, state-change rules, preservation of specifics.
- **Context-injection rubric:** **verbatim port** of [`OBSERVATION_CONTEXT_PROMPT` + `OBSERVATION_CONTEXT_INSTRUCTIONS`](https://github.com/pacifio/cersei/blob/main/bench/long-mem/src/mastra_prompts.rs) wrapped around retrieved context before it hits the answerer. Contains Mastra's `KNOWLEDGE UPDATES` / `PLANNED ACTIONS` / `MOST RECENT USER INPUT` guidance.
- **Models:** answerer, judge, and observer all on `gemini-2.5-flash`, temperature 0 (0.3 for observer, matches Mastra's `observation.modelSettings.temperature`). Embeddings: `gemini-embedding-001`, 3072-dim, with Matryoshka truncation if requested.
- **Metric:** `overall_accuracy` is the **macro average** across question types excluding abstention (matching Mastra `cli.ts:99`). Abstention is reported separately.
- **Top-k** for retrieval configs: **20** (matches Mastra's published RAG config).
- **Concurrency:** 4 in-flight questions × up to 6 observer calls per question.
- **Retry:** Gemini embedding + completion calls retry with exponential backoff (up to 6 attempts, ~30 s window) on 429 / 5xx / transport errors.
- **Security:** API keys flow through `x-goog-api-key` header, **never** in the URL query string. Error paths run any leaked URL through a redactor before logging. See [leak post-mortem](#security-hygiene) below.

## Reproduce

The runner expects `GOOGLE_API_KEY` in the environment — source from a gitignored `.env`, never commit it.

```bash

echo "GOOGLE_API_KEY=AIza..." > .env   # gitignored

  --dataset s --config all --concurrency 4 --top-k 20 \

  --extractor-model gemini-2.5-flash
```

## Security hygiene

API keys have leaked from this repo **twice** before we tightened things up — a hardcoded one in `bench/term-bench/runner-google.sh` and, later, keys embedded in reqwest error messages (`?key=…` in the URL) that ended up in tracked `.log` and `rows-*.json` files. Permanent fixes applied:

1. **Gemini calls now use `x-goog-api-key` header**, never query string — `crates/cersei-embeddings/src/gemini.rs` and `crates/cersei-provider/src/gemini.rs`.
2. **Error messages are scrubbed**: any string that could carry `key=<…>` runs through `redact_url_key` before it hits logs or result files.
3. **`.gitignore`** blocks `bench/**/*.log`, `bench/**/results*/`, `bench/**/runner-*.sh`, `bench/**/abstract-output.jsonl`, and `.env*`.
4. **Runner scripts refuse to start without `GOOGLE_API_KEY`** in env — no more inline keys.
5. **Pre-commit sanity check** (run manually): `git ls-files | xargs grep -l -E "AIza[A-Za-z0-9_-]{35}|sk-[A-Za-z0-9_-]{30,}"` must return zero files.

## Caveats

- Our `longmemeval_s` numbers are measured with `gpt-4o-mini` as both answerer and judge. Mastra's *Observational Memory* research tests several judge/answerer combinations — when comparing numbers, make sure the model pair matches. Running with `--answerer-model gpt-4o` / `--judge-model gpt-4o` is a one-line flag if you want to rerun against a different pair.
- `concurrency=4` (outer) × `INNER_EXTRACT_CONCURRENCY=6` (hybrid fact extraction) keeps us under OpenAI tier-1 rate limits. If you have higher tier throughput, bumping `--concurrency 8` will halve wall time.
- A one-shot bench like this measures recall quality with *fixed* retrieval — it does not exercise agents calling tools during answering. If you need tool-use-in-the-loop behaviour, add a `cersei::Agent` wrapper; the memory backends tested here all plug in via [`.memory(...)`](/docs/api-agent) on the builder.
